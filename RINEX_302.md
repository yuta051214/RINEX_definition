# RINEX 3.02

## 1. RINEX の理念と歴史

Receiver Independent Exchange Format（RINEX）の最初の提案は、EUREF 89 大規模 GPS キャンペーン（60台以上の異なるメーカー製受信機参加）で収集される GPS データを容易に交換できるよう、ベルン大学天文研究所により開発されました。開発時の最大の配慮点は以下の通りです。

ほとんどの測地処理ソフトウェアが共通して利用する観測値は、

  - 搬送波のキャリア位相観測（受信した衛星信号搬送波と受信機生成リファレンス周波数のビート周波数測定）
  - 擬似距離（コード）観測（受信時刻〔受信機時刻系〕と送信時刻〔衛星時刻系〕の差）
  - 観測時刻（上記キャリア位相またはコード観測が有効となった受信機クロックの読み値）

通常、観測時刻は位相・コード観測の双方および全衛星に共通と仮定されるため、ソフトウェアが必要とする受信機情報は「位相・コード・観測時刻」と局名やアンテナ高などの局情報だけで十分でした。

これまでに開発・公開された主なフォーマットは以下です。

- **RINEX Version 1**  
   - 1989年に第5回国際測地衛星測位シンポジウム（ラスクルーセス）で発表・承認
- **RINEX Version 2**  
   - 1990年に第二回精密測位シンポジウム（オタワ）で発表。GLONASSやSBASの追跡データを含める機能を追加
   - **Version 2.10**: 非整数秒サンプリング対応＋生信号強度観測追加
   - **Version 2.11**: L2C擬似距離用2文字コード定義＋GEO NAV MESS修正
   - **Version 2.20**: IGS LEOパイロット向け非公式版

また、RINEX の考え方を応用した IGS 向け交換フォーマットも策定されています。

- 衛星／受信機クロックオフセット交換フォーマット
- SBAS 放送データ交換フォーマット
- IONEX（電離圏モデル交換フォーマット）
- ANTEX（測地用GNSSアンテナ位相中心変位交換フォーマット）

将来の Galileo や拡張GPS（新周波数・観測型、多チャネル追跡）への対応強化、および複数衛星システム混在ファイルの取り扱い改善を目的として、データレコード構造を大幅に改訂し80文字制限を撤廃した **RINEX Version 3** が策定されました。  
さらに **Version 3.01** では、異なる追跡モード／チャネル間で位相整合を図るために必要に応じ1/4サイクルシフトを適用できる仕様が追加されています。





## 2. 一般フォーマットの説明

RINEX バージョン 3.00 形式は、以下の３種類の ASCII ファイルで構成されています

1. **観測データファイル（Observation data File）**  
2. **ナビゲーションメッセージファイル（Navigation message File）**  
3. **気象データファイル（Meteorological data File）**  

各ファイルは「**ヘッダセクション**」と「**データセクション**」から成り、ヘッダセクションはファイルの先頭に配置されます。  
ヘッダセクション内の各行では、**61～80 文字目**にヘッダラベルを置き、仕様書に示されたとおり正確に記述することが必須です。

フォーマットは、受信機や衛星システムごとの観測型の数に依存せずにファイルサイズを最小化できるよう設計されています。  
ヘッダで「格納する観測型」と「観測された衛星システム」を指定することで、可変長レコードを許すシステムでは観測レコードを必要最小限に短く保ち、末尾の空白を省略できます。  
観測レコードに最大長制限はありません。

観測データファイルおよび気象データファイルは基本的に「１局・１セッション分」のデータを含みます。RINEX バージョン 2 以降では、ローミング受信機が複数の局を順次占有するケース（ラピッドスタティックやキネマティック測位）にも対応可能ですが、異なる受信機（またはアンテナ）のデータを同一ファイルに連結することは推奨されません。

複数の受信機データを交換する場合、同一衛星のナビゲーションメッセージを重複して含めるのは非効率です。そこで、ある受信機から取得したナビゲーションメッセージファイルを交換するか、複数受信機の情報を統合して冗長性のないメッセージファイルを作成する方法が推奨されます。

RINEX バージョン 1 のナビゲーションメッセージファイル形式は旧 NGS 交換フォーマットと同一でしたが、バージョン 3 では GPS、GLONASS、Galileo、準天頂衛星システム（QZSS）、北斗（BDS）、SBAS の複数システムのメッセージを同一ファイルに含めることができます。

各データレコードの詳細なフォーマット説明および例は、ドキュメント末尾の付録表に掲載されています。


> **「観測型」って何？**
> 
> RINEX 3.02 における **観測型（observation types）** とは、ファイルに含める各種観測データの種類を示す **3文字のコード**を指します。  
> これらのコードは、ヘッダの **`SYS / # / OBS TYPES`** レコードで列挙される。  
>
> 具体例：  
>   - `C1C`, `L1C`, `D1C`, `S1C` ： L1 信号の各観測値
>       - `C1C` ： L1 周波数の **コード擬似距離** 
>       - `L1C` ： L1 周波数の **キャリア位相**  
>       - `D1C` ： L1 周波数の **ドップラー**  
>       - `S1C` ： L1 周波数の **信号強度** 
>   - `C2X`, `L2X`, `D2X`, `S2X` ： L2 信号の各観測値  
>   - `C5X`, `L5X`, `D5X`, `S5X` ： L5 信号の各観測値  
> 
> 各コードは以下の３要素で構成されます
>
> 1. **t（type）**：観測量の種類  
> 2. **n（number: 周波数番号）**：1=L1, 2=L2, 5=L5, …  
> 3. **a（attribute: 属性）**：チャネルや信号生成モード（C/A コード由来なら `C`，複数チャネル混合なら `X` など）  
> 
> これにより、ヘッダで定義した観測型の順序どおりに各エポック・各衛星のデータレコードが並びます





## 3. 基本定義

GPS の観測値には、定義すべき三つの基本量が含まれます

- 時刻 (Time)
- 距離 (Range)
- 位相 (Phase)

### 3.1 時刻 (Time)  

観測時刻は、受信信号を受信したときの受信機時刻です

- 位相観測と擬似距離観測の両方で同一の時刻を用います  
- 同一エポック内の全衛星に対して同一の時刻です
- 単一システムのファイルでは、デフォルトで該当衛星システムの時刻系が使用されます
- 混合システムのファイルでは、必ずヘッダの **Start Time** レコードで実際の時刻を指定する必要があります

### 3.2 擬似距離 (Pseudo-Range)  
擬似距離 (PR) とは、受信機アンテナから衛星アンテナまでの **幾何距離（distance）** に、**受信機および衛星の時計オフセット（receiver clock offset / satellite clock offset）** や **大気遅延などのバイアス（other biases）** を加えたものです。  

```text
PR = distance + c * (receiver clock offset – satellite clock offset + other biases)
```

ここで c は光速（m/s）を表します。　※ つまり定数？  
この式により、擬似距離（PR）は受信機・衛星のクロック挙動を反映します。  
ファイル中ではメートル単位で格納されます。

混合 GPS/GLONASS/Galileo/QZSS/BDS ファイルにおける擬似距離の取り扱いについては、第 8.1 章をご参照ください。





### 3.3 位相 (Phase)

位相観測値は、搬送波位相（carrier-phase）を**全周サイクル数**で表したものです。  
スクエアリング型受信機が測定する**半サイクル**は、全周サイクル数に変換し、対応する観測コードでフラグを立てる必要があります（GPSのみ、Table 2および§5.4参照）。

位相の変化は擬似距離の変化と同符号（負のドップラー）です。  
隣接エポック間の位相観測は、**整数サイクル数**を足し合わせて継続性を保つ必要があります。

位相観測値には、大気屈折や衛星時計オフセットなどの**外部補正**は行われません。

必要に応じて、異なる信号チャネル間で同一周波数・同一衛星システムの位相を一致させる**位相シフト補正**を適用します（§9.1および付録A19参照）。

受信機または変換ソフトがリアルタイム算出の **受信機時計オフセット dT(r)** を用いて観測値を補正する場合、位相／擬似距離／エポックの３量の整合性を保つため、以下のようにすべての観測量に同じ時計補正 dT(r) を適用します。

  ```text
  Time (corr)  = Time(r)   – dT(r)
  PR   (corr) = PR(r)     – dT(r) × c
  phase (corr)= phase(r)  – dT(r) × freq
  ```

> (corr)：correctedの略、補正後の意味  
> (r)： receiverの略、受信機の意味  
> c：光速  
> freq：搬送波の周波数（L1, L2, L5 など）





### 3.4 ドップラー （Doppler）  
追加の観測量としてのドップラーシフトの符号は通常どおり定義され、**衛星が接近するときに正**となります。


### 3.5 衛星番号 （Satellite numbers）  
RINEX Version 2 以降では、従来の二桁番号 nn の前に「衛星システム識別子 s」1文字を付加した「S NN」で衛星を識別します。

衛星番号とコンステレーション識別子（S NN）：

- **S（システム識別子）**：  
    - G：GPS  
    - R：GLONASS  
    - S：SBAS ペイロード  
    - E：Galileo  
    - C：BeiDou  
    - J：QZSS 
- **NN**：  
    - GPS／Galileo／BeiDou　=>　PRN  
    - GLONASS　=>　スロット番号  
    - SBAS, QZSS (L1 – SAIF)　=>　PRN − 100  
    - QZSS (その他)　=>　PRN − 192  

同じシステム識別子は、ヘッダの各種レコード中でも必要に応じて用いられます。  


> 衛星番号フィールドは２桁の “NN” 部分に、元の衛星識別番号をうまく収めるためにシステムごとに異なる「オフセット引き算」を行っています。具体的には次のようなルールです。
>
> 	1. GPS／Galileo／BeiDou
> 	    - これらは元々の PRN 番号（1～32など）をそのまま “NN” に使います。
> 	    - 例：GPS の PRN 7 → G07
> 	2. GLONASS
> 	    - 衛星に付与された「スロット番号」（1～24）を “NN” に使います。
> 	    - 例：GLONASS のスロット 5 → R05
> 	3. SBAS と QZSS（L1-SAIF モード）
> 	    - SBAS（WAAS, EGNOS など）は PRN が 120～158 程度の 100 番台になります。
> 	    - QZSS の L1-SAIF も同様に 100 番台 PRN を使うため、二桁の NN に収めるには 100 を引く 形にします。
> 	    - たとえば PRN 120（SBAS）→ 120 − 100 = 20 → S20
> 	    - たとえば PRN 193（QZSS-SAIF）→ 193 − 100 = 93 → J93
> 	4. QZSS（その他モード）
> 	    - QZSS の通常モード（L1C, L2C, L5 など）は PRN が 192～198 番台になります。
> 	    - これも二桁にするために 192 を引く 形で NN を算出します。
> 	    - たとえば PRN 193 → 193 − 192 = 1 → J01
> 	    - たとえば PRN 196 → 196 − 192 = 4 → J04
> 
> こうすることで、どのシステムのどの衛星かを示しつつ、常に２桁の番号 sNN に収められるようになっています。





## 4. RINEX ファイルの交換

オリジナルの RINEX ファイル命名規則は、MS-DOS 時代の「8.3 文字」制限下で策定されました。  
現代の OS では最大 255 文字までサポートされるため、RINEX 3.02 ではより 記述的・柔軟・拡張性 を高めた新しい命名規則を導入しています。  
以下の表は推奨されるファイル名パラメータを示します。

`Name_S_Start_Time_Period_ObsFreq_Content.Format(.Compression)`

**Figure 2: 推奨されるファイル名パラメータ**  

| フィールド                         | 文字数 | 説明                                                               |
|------------------------------------|------:|--------------------------------------------------------------------|
| **Name**                           |     9 | サイト／ステーション／国コード                                     |
| **S** (Data Source)                |     1 | データソース識別子                                                 |
| **Start Time**                     |    11 | 開始時刻（YYYYDDDHHMM）                                            |
| **Period**                         |     3 | ファイル収録期間（名目値）                                         |
| **Obs. Freq.**                     |     3 | 観測インターバル                                                  |
| **Content**                        |     2 | ファイル内容タイプ                                                 |
| **Format**                         |     3 | ファイルフォーマット                                               |
| **Compression** (任意)             |  2–3 | 圧縮方式識別子                                                     |

各要素は固定長で、アンダースコア `_` で区切ります（`Format` と `Compression` はドット `.` で区切り）。  
フィールドはゼロ埋めして桁幅を満たす必要があります。  
`Compression` フィールドは省略可能です。  
詳細は付録 A1 を参照してください。

**Table 1: サンプルファイル名とコメント**

| ファイル名                                                       | コメント                                                     |
|------------------------------------------------------------------|-------------------------------------------------------------|
| `ALGO00CAN_R_20121601000_01H_01S_MO.rnx`                         | Mixed RINEX GNSS 観測ファイル（1時間分、1秒間隔）              |
| `ALGO00CAN_R_20121601000_15M_01S_GO.rnx`                         | GPS RINEX 観測ファイル（15分分、1秒間隔）                           |
| `ALGO00CAN_R_20121601000_01H_05Z_MO.rnx`                         | Mixed RINEX GNSS 観測ファイル（1時間分、5Hz 追跡）             |
| `ALGO00CAN_R_20121601000_01D_30S_GO.rnx`                         | GPS RINEX 観測ファイル（1日分、30秒間隔）                           |
| `ALGO00CAN_R_20121601000_01D_30S_MO.rnx`                         | Mixed RINEX GNSS 観測ファイル（1日分、30秒間隔）              |
| `ALGO00CAN_R_20121600000_01D_GN.rnx`                             | GPS RINEX ナビゲーションファイル（1日分）                          |
| `ALGO00CAN_R_20121600000_15M_RN.rnx`                             | GLONASS RINEX ナビゲーションファイル（1日分）                      |

---

観測ファイルサイズをさらに削減するため、畑中勇気氏が提案した **差分圧縮方式**（Hatanaka Compression Scheme）では、同一衛星・同一観測型の時系列データ間で高次差分を取り、その結果を ASCII 形式ファイルにまとめます。  
これをさらに標準的な圧縮プログラムで圧縮して配布します。  

方式の詳細：  
- http://terras.gsi.go.jp/ja/crx2rnx.html  
- IGS Mails 1525, 1686, 1726, 1763, 1785, 4967, 4969, 4975  

なお、ファイル命名および圧縮方式の推奨事項は厳密には RINEX フォーマット定義の一部ではありませんが、IGS など大規模なユーザコミュニティでのデータ交換を大幅に容易にします。  

> **ここでいう mixed とは複数のGNSSからのデータを含んでいるといった意味でしょうか？**  
>
> はい、そのとおりです。ここでの「mixed（M）」は、複数の衛星システム（たとえば GPS、GLONASS、Galileo、QZSS、BDS など）の観測データを同一ファイルに含んでいることを意味します。 
> 
> ファイル名の「M O」部分では、 
> 
>   - M＝Mixed（混在）
>   - O＝Observation（観測データファイル）  
>
> を表しています。





## 5. RINEX Version 3 の新機能

以下に示すのは、RINEX Version 3 で導入された主な新機能です。

### 5.1 観測コードの拡張

GPS、Galileo、BDS の新しい信号構造により、単一または複数チャネルの組み合わせに基づくコード／位相観測が可能になりました。  
- 2チャネル信号：I 成分 + Q 成分  
- 3チャネル信号：A 成分 + B 成分 + C 成分  
- E5a＋E5b の広帯域トラッキングもサポート  

観測コードを短く保ちつつ信号生成の詳細を示せるよう、従来の 2 文字から 3 文字に拡張し、末尾に「信号生成属性」を追加しています。  

- **形式**：`tna`  
  - **t**：観測タイプ  
    - C = 擬似距離 (Pseudo‐range)  
    - L = キャリア位相 (Carrier‐phase)  
    - D = ドップラー (Doppler)  
    - S = 信号強度 (Signal‐strength)  
  - **n**：周波数バンド番号 (1…8)  
  - **a**：属性／トラッキングモードやチャネル (I, Q, A, B, C, X など)  

#### 例
- **L1C**：GPS/GLONASS の C/A コード由来 L1 キャリア位相観測  
- **C2L**：GPS の L チャネル由来 L2C 擬似距離観測

### 5.2 衛星システム依存観測型リスト  
観測レコード内で各エポック・各衛星ごとに格納される観測値の並び順は、ヘッダレコードの観測コード一覧（`SYS / # / OBS TYPES`）によって決まります。  
受信機が出力する観測型は衛星システムごとに大きく異なる可能性があるため、RINEX Version 3 ではシステム依存の観測コードリストを必ず記述することを要求しています。

> **「衛星システム依存観測型リスト」の仕様に基づく、各システムの SYS / # / OBS TYPES ヘッダ行例：**
>
> 例1: GPS 単一周波数（L1 C/A コードのみ）  
> `G   4  C1C  L1C  D1C  S1C      SYS / # / OBS TYPES`
> 
> 例2: GPS デュアルバンド（L1 C/A + L2C）  
> `G   8  C1C  L1C  D1C  S1C  C2C  L2C  D2C  S2C   SYS / # / OBS TYPES`
> 
> 例3: GLONASS 単一周波数（G1 C/A コードのみ）  
> `R   4  C1C  L1C  D1C  S1C      SYS / # / OBS TYPES`
> 
> 例4: Galileo E1 B チャネルのみ  
> `E   4  C1B  L1B  D1B  S1B      SYS / # / OBS TYPES`
> 
> 例5: QZSS 三周波数（L1 C/A, L2C, L5I）  
> `J  12  C1C  L1C  D1C  S1C  C2L  L2L  D2L  S2L  C5I  L5I  D5I  S5I   SYS / # / OBS TYPES`
>
> 各行の先頭１文字（G/R/E/J）は衛星システムコード。  
> ２番目の数値はそのシステムで定義する観測型の数。  
> 以降がヘッダに並べる観測コードのリストで、実際のデータレコードもこの順に並びます。  

> **ここでいう「観測型」とは、observation type を翻訳したものと思いますが、同じくこのpdf中で用いられている observation code と同一のものでしょうか？**
> 
> はい。そのとおりです。  
> RINEX 3.02 では「観測型（observation types）」という概念を、ヘッダの SYS / # / OBS TYPES で列挙しますが、ファイル中に実際に書き込まれるのはそれぞれを表す 3文字の“観測コード（observation codes）” です。  
> つまり「観測型」を定義する際には、必ずその型を識別する「観測コード」を用い、「C1C」「L1C」などのコード一覧がまさに観測型のリストを構成しています。  
> 
> 言い換えると、
> 
> “observation type” ＝ “observation code”
> 
> であり、PDF 中で observation codes と呼ばれているものが、そのまま列挙された観測型（＝observation types）を指しています。





### 5.3 マーカ種別 (Marker Type)  
マーカ（測地点・アンテナ設置位置）の性質を示すため、`MARKER TYPE` ヘッダレコードを定義しています。  
`GEODETIC` と `NON_GEODETIC` を除き、すべてのマーカタイプでこのレコードが**必須**です。

| Marker Type    | 説明                                    |
|---------------|-----------------------------------------|
| **GEODETIC**     | 大地固定型高精度モニュメント             |
| **NON_GEODETIC** | 大地固定型低精度モニュメント             |
| **NON_PHYSICAL** | ネットワーク処理によって生成             |
| **SPACE_BORNE**  | 軌道上宇宙機                             |
| **AIR_BORNE**    | 航空機、気球など                         |
| **WATER_CRAFT**  | 船舶                                   |
| **GROUND_CRAFT** | 陸上移動体                             |
| **FIXED_BUOY**   | 水上に“固定”されたブイ                 |
| **FLOATING_BUOY**| 水上に浮かぶブイ                       |
| **FLOATING_ICE** | 浮氷                                   |
| **GLACIER**      | 氷河上に“固定”                         |
| **BALLISTIC**    | ロケット、弾道ミサイルなど             |
| **ANIMAL**       | 動物に搭載された受信機                 |
| **HUMAN**        | 人が携行                               |

- `GEODETIC` ／ `NON_GEODETIC` は必須ではありません。  
- これ以外のマーカタイプを指定すると、「移動する受信機」でデータを収集したことを示し、ヘッダに “開始移動 (Event flag 2)” レコードを挿入する必要がなくなります。  
- ただし、複数の大地固定モニュメントを巡る場合の静止⇔運動切り替えには Event flag 2（開始移動）および 3（新サイト占有）のフラグが引き続き必要です。  
- プロジェクト固有のキーワードを追加定義することも可能です。

> marker type について、電子基準点はどれに該当しますか？
> 
> 電子基準点（全国電子基準点）は、大地に固定された高精度モニュメントにあたるので、MARKER TYPE は GEODETIC を指定します（省略可能）。
> - GEODETIC：Earth-fixed high-precision monument（大地固定型高精度モニュメント）




### 5.4 半波長観測（Half-wavelength observations） および 半周期曖昧性（half-cycle ambiguities）  
- **半波長観測**（コーデレススクエアリング技術で取得）には専用の観測コードがあります。  
- 旧版で用いていた「特別波長係数ヘッダ行」と LLI フラグ Bit 1 は不要になりました。  
- 受信機がファイル内でスクエアリング追尾⇔全周期追尾を切り替えた場合は、両タイプの観測コードを `SYS / # / OBS TYPES` ヘッダに含める必要があります。  
- 半波長観測値は「全周期サイクル数」として格納しますが、解決時には半波長を考慮する必要があります。  
- 全周期追尾でありながら LoL（ロス・オブ・ロック） Bit 1 がセットされた観測は、半周期曖昧性を示すフラグ付きとして扱います（付録 A3 参照）。





### 5.5 スケールファクタ (Scale Factor)  
オプションの `SYS / SCALE FACTOR` レコードを使うと、例えば位相データを**あらかじめ 10 倍**してからファイルに格納し、小数第4位（0.0001 サイクル）までの分解能を実現できます。  
- Ver 2.20 の `OBS SCALE FACTOR` レコードを改良したものです。  
- 10 倍、100 倍…など分解能向上のために利用します。





### 5.6 車両搭載受信機に関する情報  
車両や航空機、宇宙機搭載の受信機データ処理向けに、以下の情報を専用ヘッダレコードで追加できます。各データは**同一の車体固定座標系**で与え、姿勢情報は別途アティチュードファイルで提供します。

- **ANTENNA: DELTA X/Y/Z**  
  アンテナ参照点の車体固定座標系位置オフセット  
- **ANTENNA: B.SIGHT X/Y/Z**  
  アンテナ軸の GNSS 衛星方向単位ベクトル（地上設置アンテナの鉛直軸に対応）  
- **ANTENNA: ZERODIR X/Y/Z**  
  アンテナのゼロ方向。方位依存の位相中心変動モデル適用用  
- **CENTER OF MASS: X/Y/Z**  
  車体（宇宙機）の質量中心位置  
- **ANTENNA: PHASECENTER X/Y/Z**  
  平均位相中心位置





### 5.7 信号強度  

RINEX データレコード中の信号強度指標 sn_rnx（1＝非常に弱い…9＝非常に強い）は、生のキャリア対雑音比 sn_raw（dB-Hz）を以下の式で 1–9 の整数に標準化して得られます：  

```txt
sn_rnx = MIN( MAX( INT(sn_raw / 6), 1 ), 9 )
```

#### Table 9: 標準化された S/N 指標  

| S/N 比 (dB-Hz)  | RINEX 信号強度指標 (sn_rnx) |
|:---------------:|:---------------------------:|
| < 12            | 1                           |
| 12 – 17         | 2                           |
| 18 – 23         | 3                           |
| 24 – 29         | 4                           |
| 30 – 35         | 5                           |
| 36 – 41         | 6                           |
| 42 – 47         | 7                           |
| 48 – 53         | 8                           |
| ≥ 54            | 9                           |

なお、生のキャリア対雑音比（sn_raw）はオプションで `Sna` 観測型として dB-Hz 単位で格納でき、`SIGNAL STRENGTH UNIT` ヘッダレコードで単位を指定します。





### 5.8 PGM / RUN BY / DATE ヘッダレコードの日付／時刻形式  

ファイル生成時刻を記録する第２ヘッダ行 `PGM / RUN BY / DATE` の日時書式は次のとおり定義されました：  

```txt
yyyymmdd hhmmss zone
```

- `zone` はタイムゾーンを示す3～4文字コード  
- 推奨タイムゾーン：UTC  
- ローカル時刻を使用した場合はタイムゾーン不明を示す `LCL` を指定  





### 5.9 アンテナ位相中心ヘッダレコード  

高精度測位のため、外部アンテナ情報なしにアンテナ平均位相中心位置を記録できるオプションヘッダ `ANTENNA: PHASECENTER` を定義。  
- 周波数／衛星システムごとに、アンテナ参照点 (ARP) からの平均位相中心オフセットを指定  
- 車両搭載時は車体固定座標系でのオフセット `ANTENNA: DELTA X/Y/Z` と併用可能  
- 位相中心揺らぎ補正は §5.15 および付録 A19 を参照  





### 5.10 アンテナ方向ヘッダレコード  

傾斜設置アンテナや車載アンテナの指向性を記録するヘッダを定義：  
- **垂直軸方向（ボアサイト）**：`ANTENNA: B.SIGHT X/Y/Z`  
- **ゼロ方向（基準方向）**：`ANTENNA: ZERODIR X/Y/Z`  
これにより、設置向き依存の位相中心変動モデル（§6.14）を適用できます。  





### 5.11 観測データレコード  

従来の観測コード定義に加え、RINEX フォーマットで最も目立つ変更点は観測レコードです。衛星システムごとに異なる観測型とその並び順はすべてヘッダの `SYS / # / OBS TYPES` で指定されるため、プログラムや人間がデータを読みやすくなりました。  

- 各観測レコードは衛星番号（snn）で始まり、エポックレコードは行頭に `>` が付きます。  
- 破損ファイルやリアルタイムストリーミング時も、次のエポックレコードを容易に検出できるようになりました。  
- 旧版の 80 文字制限は撤廃され、可変長レコードを許容します。

### 5.12 電離層遅延を擬似観測値として追加

実観測値に付随する情報として、電離層モデルから得られた電離層位相遅延を「擬似観測値」として I コードで追加できます。
- t = I (Ionosphere phase delay)
- n = 1…8（周波数番号）
- a = （空欄）

各衛星システムの観測型リストに必ず１つだけ含め、以下の手順で他の波長や擬似距離補正に利用します：

```txt
corr_phase(fi)   = raw_phase(fi) + d_ion(fi)  
corr_prange(fi)  = raw_prange(fi) - d_ion(fi) * c/fi  
d_ion(fk)        = d_ion(fi) * (fi/fk)^2   # 1次効果のみ考慮  
```

- d_ion(fi): 周波数 fi に対する電離層位相補正（サイクル）
- c: 光速 (m/s)





### 5.13 チャネル番号を擬似観測値として追加

特殊用途で、各衛星に割り当てられた受信機チャネル番号を RINEX に含めることができます：  
- t = X (Receiver channel number)
- n = 1
- a = （空欄）
- 最小チャネル番号は 1（必要に応じ事前に再番号化）。
- 複数チャネル使用時は、各チャネル番号を「2桁右寄せ」で固定幅 F14.3 フィールド内に連結。順序は観測型リスト順と同じ。
- 最大 5 チャネル、番号 < 100 の制限あり。

例
- 0910.000 → チャネル 9 と 10
- 010203.000 → チャネル 1、2、3





### 5.14 差分コードバイアス (DCB) 補正

高精度用途向けに、差分コードバイアスを適用済みの RINEX ファイルを生成可能です。
- 例：cc2noncc（J. Ray 2005）
- 適用済みファイルを示すヘッダ：SYS / DCBS APPLIED





### 5.15 アンテナ位相中心変動(PCV)補正

高精度向けに、アンテナの仰角・方位依存 PCV モデルを適用済みの RINEX ファイルを生成可能です。
- 適用済みファイルを示すヘッダ：SYS / PCVS APPLIED





### 5.16 ナビゲーションメッセージファイル
- すべての衛星システムでヘッダ形式を統一。
- データ部では各メッセージブロックの最初に衛星番号に加えシステム識別子を追加。
- システム依存のヘッダレコードにも同様の識別子を含め、該当システムごとに繰り返します。

例

```txt
G06 1999 09 02 17 51 44 -.839701388031D-03 -.165982783074D-10 .000000000000D+00
GPSA .1676D-07 .2235D-07 .1192D-06 .1192D-06 IONOSPHERIC CORR
GPSB .1208D+06 .1310D+06 -.1310D+06 -.1966D+06 IONOSPHERIC CORR
GAL  .1234D+05 .2345D+04 -.3456D+03 IONOSPHERIC CORR

# 各行でシステム識別子（G, GPSA, GPSB, GAL）が先頭に付加される
```





## 6. 追加のヒントと注意事項

### 6.1 バージョン確認  
RINEX ファイルを読み込むプログラムは、必ずファイル内のバージョン番号を検証してください。  
たとえ新機能を使っていなくとも、新しいバージョンのファイルは見た目が変わっている可能性があります。

### 6.2 文字列フィールドの先頭空白  
- **読み込み側**：CHARACTER 型フィールドから先頭の空白文字を自動的に削除することを推奨します。  
- **書き込み側**：CHARACTER 型フィールドには左詰め（left-justify）で変数を配置してください。

### 6.3 可変長レコード  
- ASCII ファイルは通常レコード長が可変なので、まずは十分長い空文字列バッファに観測レコード全体を読み込み、その後でデータを解析することを推奨します。  
- 可変長レコードでは、末尾の空フィールド（特にオプションの受信機クロックオフセット）が省略されることがあります。

### 6.4 空白フィールドの扱い  
将来の拡張に備え、現在「空欄」と定義されているフィールド（フォーマット指定 nX）は**スキップせずに**丁寧に読み飛ばすよう実装してください。将来、ここに新しい内容が割り当てられる可能性があります。

### 6.5 ヘッダ／データレコードの順序  
- ヘッダ各行の 61–80 文字目にあるレコード識別子（descriptor）は必須です。読み込みプログラムは、まず内部バッファにヘッダを読み込んだ上で、識別子に応じた書式でパースしてください。  
- ヘッダレコードの順序は自由ですが、以下は例外として固定してください：  
  1. `RINEX VERSION / TYPE` はファイルの先頭行であること  
  2. `SYS / # / OBS TYPES` は `SYS / DCBS APPLIED` や `SYS / SCALE FACTOR` よりも前に置くこと  
  3. `# OF SATELLITES` レコード（存在する場合）は直後に対応する `PRN / # OF OBS` レコードを続けること（ドキュメント用途のためオプション）  
  4. `END OF HEADER` はヘッダの最終行  

#### データレコード  
- **同一時刻タグ**を持つ複数のエポックデータレコードは許可されません（例外：イベントレコード）。  
- エポック（時刻）は**時系列順**に並べる必要があります。  





### 6.6 欠損項目と値の有効期間  
ファイル作成時に不明な項目は、ゼロまたは空欄とするか、該当レコード自体を省略して構いません。  
そのため、欠損したヘッダレコードの項目は、読み込みプログラム側でゼロまたは空欄として扱われます。  
レコード末尾の空白は切り詰めても構いません。  
各値は、追加のヘッダレコードで上書きされるまで有効となります。





### 6.7 未知／未定義の観測型およびヘッダレコード  
RINEX ファイルを読み込むプログラムでは、未知の観測型やヘッダレコード、イベントフラグを適切にスキップしたり、ユーザーに通知したりする実装が望ましいです。  
また、ヘッダのバージョン番号を必ずチェックし、対応できない場合は適切にエラー処理を行ってください。





### 6.8 イベントフラグレコード  
「衛星数 (# of satellites)」は、エポックレコード直後に続く同一エポックの観測レコード数とも一致します。  
そのため、特定のイベントフラグを詳細に評価しない場合は、この数を使って後続レコードをまとめてスキップできます。





### 6.9 受信機クロックオフセット  
RINEX 観測ファイルでは、オプションで受信機由来のクロックオフセットを記録できます。  
データ（エポック、擬似距離、位相）がこのオフセットで補正済みか否かを明確にするため、バージョン 2.10 以降では **`RCV CLOCK OFFS APPL`** ヘッダレコードを必ず設けるよう要求しています。  
これにより、必要に応じて元の未補正観測値を再構築できます。





### 6.10 2桁年  
- RINEX Version 2 では、データレコードの年を2桁で格納しています。  
- 観測ファイルヘッダには `TIME OF FIRST OBS` で4桁年を、GPSナビファイルにはGPS週番号を含むため、これらから正しい西暦年を一意に復元可能です。  
- 気象データや GLONASS/GEO ナビメッセージでは100年のあいまい性があるため、2桁年は以下のように解釈してください：  
  - **80–99** → 1980–1999  
  - **00–79** → 2000–2079  
- RINEX Version 3 以降では、すべて4桁年を正式に定義します。





### 6.11 フィット間隔（GPSナビゲーションメッセージファイル）  
サブフレーム2 のワード10 ビット17 は「フィット間隔」フラグで、GPSコントロールセグメントが軌道パラメータ算出に用いたフィット間隔を示します（ICD‐GPS‐200 §20.3.3.4.3.1参照）：  
- **0** → 4時間  
- **1** → 4時間超  

このフラグ値と IODC、および ICD‐GPS‐200 の表20-XII を組み合わせることで、実際のフィット間隔（時間）を算出できます。  
各メッセージブロックの最後のレコード第2フィールドには、IODC・フィットフラグ・表20-XIIから決定したフィット間隔（時間）が格納されます。  





### 6.12 衛星健全度（GPSナビゲーションメッセージファイル）  
サブフレーム1 のワード3 ビット18–22 で報告される各信号成分の健全度情報は、Version 2.10 以降、ナビゲーションメッセージ第6レコードの第2フィールドに格納されます。  
読み込みプログラムは以下の RINEX 値を見れば、ビット17 のみ格納されているか（単一ビット）あるいはビット18–22 まで全て格納されているかを判断できます。

| RINEX 値      | 意味                                           |
|--------------:|-----------------------------------------------|
| 0             | 健全（Health OK）                              |
| 1             | 健全ではない（ビット18–22 は未格納）           |
| >32           | 健全ではない（ビット18–22 が全て格納されている）|

---

#### 6.13 メッセージ送信時刻（GPSナビゲーションメッセージファイル）  
メッセージの送信時刻は、土曜日深夜から日曜日未明にかけて、週の切替時刻直前・直後で報告されることがあります。  
RINEX の Nav メッセージ中の週番号（`BROADCAST ORBIT -5` レコードに含まれる）は “Time of Ephemeris (ToE)” に対応し、これは元の GPS 衛星メッセージ内の週番号とは異なります。  
同じ週に属することを明示するため、送信時刻は **604 800 秒（1 週間）減算** して、負の値として表現します。

#### 6.14 アンテナ参照点と位相中心  
- **マーカ**：アンテナが強制センタリングまたは三脚で設置された「測地参照モニュメント」。  
- **アンテナ参照点 (ARP)**：アンテナ本体の明確に定義された点（例：プリアンプ底面中心）。マーカから ARP までの高さは `ANTENNA: DELTA H/E/N` で報告。  
  - 微小な水平オフセットは同レコードで併記可能。  
  - 車両搭載の場合は、車体固定座標系で `ANTENNA: DELTA X/Y/Z` によって報告。  
- **平均位相中心**：周波数依存かつ最低仰角依存の ARP 上方の平均位相中心位置。  
  - 混在アンテナネットワークで重要。  
  - 絶対位置または参照アンテナ比で、`ANTENNA: PHASECENTER` ヘッダで指定。  
  - 固定局では北/東/上方向、車載では車体固定 X/Y/Z。  
  - より高精度用途は仰角または方位依存の PCV モデル適用（§5.15／付録 A19）。  
- **アンテナの向き**：  
  - **ゼロ方向**：固定局は通常「北向き」。`ANTENNA: ZERODIR AZI`（方位角）または `ANTENNA: ZERODIR XYZ`（車体固定単位ベクトル）で指定。  
  - **ボアサイト**：車載では GNSS 衛星方向のアンテナ対称軸。`ANTENNA: B.SIGHT XYZ` で車体固定単位ベクトルを指定。  
  - 傾斜固定局も左手系北/東/上で同様に報告可能。  

これらを正しく解釈するには、`MARKER TYPE` レコードがヘッダに含まれていることが重要です。

#### 7. アンチスプーフィング (AS) 下の RINEX  
- 一部受信機は AS 有効時にクロスコリレーション技術で第一周波数と第二周波数のコード遅延差を算出し、L2 の位相観測を全サイクルで復元します。  
- L1 の C/A コード遅延と観測差から、第二周波数のコード遅延観測を生成可能です。他機種は Y コードを分解して P/W コード観測を復元します。  
- これらの観測はノイズレベルが増大しやすいため、RINEX v2 では LLI ビット2（値を＋4）でフラグ付けしていました。  
- RINEX v3 では、観測型コードに **codeless, semi-codeless, Z-tracking** などの属性を持たせることで AS 感染観測を詳細に表現し、従来の AS フラグは不要となりました。
